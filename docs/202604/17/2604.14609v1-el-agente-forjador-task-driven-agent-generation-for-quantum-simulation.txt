AI for science promises to accelerate the discovery process. The advent of large language models (LLMs) and agentic workflows enables the expediting of a growing range of scientific tasks. However, most of the current generation of agentic systems depend on static, hand-curated toolsets that hinder adaptation to new domains and evolving libraries. We present El Agente Forjador , a multi-agent framework in which universal coding agents autonomously forge, validate, and reuse computational tools through a four-stage workflow of tool analysis, tool generation, task execution, and iterative solution evaluation. Evaluated across 24 tasks spanning quantum chemistry and quantum dynamics on five coding agent setups, we compare three operating modes: zero-shot generation of tools per task, reuse of a curriculum-built toolset, and direct problem-solving with the coding agents as the baseline. We find that our tool generation and reuse framework consistently improves accuracy over the baseline. We also show that reusing a toolset built by a stronger coding agent can reduce API cost and substantially raises the solution quality for weaker coding agents. Case studies further demonstrate that tools forged for different domains can be combined to solve hybrid tasks. Taken together, these results show that LLM-based agents can use their scientific knowledge and coding capabilities to autonomously build reusable scientific tools, pointing toward a paradigm in which agent capabilities are defined by the tasks they are designed to solve rather than by explicitly engineered implementations.

## 1 Introduction 

Automating scientific research has long been a central aspiration of the scientific community. Large language models (LLMs) (Brown et al., 2020; Achiam et al., 2023; Jablonka et al., 2024; Anthropic, 2024) and LLM-based agents (Hong et al., 2023; Wang et al., 2024; Anthropic, 2025) have brought this vision closer to reality, demonstrating emergent physical and chemical reasoning (Moreno et al., 2026; M. Bran et al., 2024), autonomous scientific research capabilities (Boiko et al., 2023; Lu et al., 2026; Pérez-Sánchez et al., 2026), and sophisticated reasoning about experiment execution (Cao et al., 2025; Yoshikawa et al., 2023; Darvish et al., 1

2024; Song et al., 2025). Building on these advances, domain-specific scientific agents now achieve strong performance on specialized tasks through expert-designed tools and workflows (Zou et al., 2025; Wang et al., 2025; Pham et al., 2025). Our group has developed several agents of this type under the El Agente umbrella which span quantum chemistry (Zou et al., 2025; Pérez-Sánchez et al., 2026; Bai et al., 2026), quantum dynamics simulations (Gustin et al., 2025), solid-state simulations (Kumar et al., 2026), and molecular structure manipulation and editing (Choi et al., 2026). However, these systems require substantial human effort to design, configure, and maintain: each new domain or computational method necessitates significant engineering to adapt existing systems or build new ones from scratch. The root cause is that traditional agent architectures treat their toolsets as static components. Tools must be manually implemented and integrated before deployment, and when new techniques emerge, software libraries evolve, or research questions shift to adjacent domains, human developers must intervene. This rigid paradigm creates a critical bottleneck: the pace of scientific agent development cannot keep up with the diversity and evolution of scientific research needs. What is needed is an architecture in which agents can autonomously generate, validate, organize, and compose tools, so that the toolset co-evolves with the research problems it serves. We introduce El Agente Forjador (“The Forging Agent”), a multi-agent system that automates not only scientific problem-solving but also the generation of the specialized tools it requires (see Figure 1). Rather than relying on a fixed toolkit, El Agente Forjador dynamically creates, tests, refines, and composes tools in response to task requirements, adapting to new challenges without human intervention in tool generation. The system is organized around a four-stage agentic workflow, namely ( tool analysis , tool generation , task execution , and solution evaluation ), which iterates until all task criteria are satisfied. Our approach builds on the inherent power of coding agents (Team et al., 2025; OpenAI, 2025; Anthropic, 2025), which, designed for code development, have several features that can aid scientific discovery. The framework introduces several key architectural choices. All subagents are universal : every agent can generate scripts, execute computations, debug errors, and refine existing tools, enabling robust error recovery through 2mutual correction. A progressive disclosure mechanism and a dedicated toolset optimizer keep the growing library navigable by exposing only task-relevant tools and reorganizing tools into hierarchical categories. Finally, an automated solution evaluator drives iterative refinement by verifying bug-free execution, script completeness, simulation adequacy, and result completeness after each run. We evaluate El Agente Forjador on two diverse benchmarks: (i) 13 undergraduate-level quantum chemistry tasks (Zou et al., 2025; Bai et al., 2026) covering molecular structure optimization and thermochemistry & reactivity, and (ii) 11 quantum dynamics tasks (Gustin et al., 2025) spanning many-body quantum dynamics and quantum state preparation & control. 

Key Contributions. 

• We present El Agente Forjador , a multi-agent framework that autonomously solves undergraduate-level quantum chemistry (Zou et al., 2025; Bai et al., 2026) and quantum dynamics (Gustin et al., 2025) tasks by generating tools for these domains. We show that autonomous tool generation maintains or increases task accuracy relative to tool-free baselines, but incurs higher API cost due to the overhead of tool creation and refinement. 

• We show that curriculum learning and tool reuse effectively mitigate this cost: reusing curriculum-built tools, underpinned by progressive disclosure, reduces API cost by 33% to 78% and wall-clock time by up to 88% relative to zero-shot tool generation, while maintaining or improving accuracy. Because the toolset is built on a stronger model (Claude Opus 4.6) and reused by weaker ones, this constitutes a form of strong-to-weak knowledge transfer through tool interfaces: weaker models achieve up to 16.5 percentage points higher task success rate by composing well-tested tools that encode correct domain methodology. 

• We demonstrate cross-domain composability through two case studies: tools originally forged for quantum chemistry and quantum dynamics are composed to solve hybrid quantum-chemistry/quantum-dynamics tasks. This points toward a future in which scientific agents with broad, multi-domain capabilities can be assembled by composing tools forged across disciplines by El Agente Forjador .

## 2 Related Work 

Scientific Agents Recent benchmarks confirm that frontier LLMs already possess strong scientific knowledge. For example, Claude Opus 4.6 (Anthropic, 2026) achieves 53.0% accuracy on the Humanity’s Last Exam (HLE) (Phan et al., 2025), which is a collection of expert-level questions spanning science, mathematics, and engineering that were deemed very hard for LLMs to solve. LLM-based scientific agents have been built on this foundation to automate scientific tasks. In quantum chemistry, El Agente Q (Zou et al., 2025) and El Agente Quntur (Pérez-Sánchez et al., 2026) use a hierarchical memory framework to decompose natural language prompts into executable simulation tasks. In quantum dynamics, El Agente Cuántico (Gustin et al., 2025) extends this paradigm to quantum simulation workflows spanning open-system dynamics and tensor-network methods. For solid-state simulations, El Agente Sólido (Kumar et al., 2026) translates high-level scientific objectives into Quantum ESPRESSO (Giannozzi et al., 2009) pipelines integrating density functional theory ( DFT ), phonon calculations, and machine-learning potentials, while DREAMS (Wang et al., 2025) pairs a central LLM planner with specialized subagents for structure generation, DFT convergence testing, and high-performance computing ( HPC ) scheduling. El Agente Gráfico (Bai et al., 2026) takes a complementary approach by encoding scientific workflows as type-safe execution graphs backed by dynamic knowledge graphs, separating the execution layer from LLM reasoning to reduce token cost and hallucination. Beyond chemistry, k-agents (Cao et al., 2025) automates quantum computing laboratory experiments, and Lu et al. (2026) demonstrates automation of machine learning research: the system generates research ideas, writes and executes experiments, and produces full papers that can pass peer-review by humans. Despite their success, most of these systems share a common constraint: their toolsets must be updated manually whenever new tasks, methods, or software versions arise.

This capability has been composed into coding agents —systems that pair an LLM with terminal access and file-system interaction to autonomously write, execute, and debug code (Dong et al., 2025). Multiple coding-agent frameworks have emerged: Claude Code (Anthropic, 2025), Codex (OpenAI, 2025), Cursor (Anysphere, 2023), and OpenCode (OpenCode.ai, 2025). Given the scientific knowledge of LLMs, coding agents are well-suited for scientific automation. They can reason about domain-specific computations while retaining their general-purpose coding and debugging capabilities. 

Evolving Agents and Tool Creation A parallel line of research investigates agents that can extend their own capabilities by modifying themselves to achieve self-evolution. The self-evolving agent paradigm reframes agents as systems that iteratively update their memories, toolset, and architectures based on trajectory feedback to maximize long-term utility (Gao et al., 2025). Early tool-creation systems such as CRAFT and CREATOR used LLM prompting to synthesize new tools, with CRAFT discarding broken tools via execution feedback (Yuan et al., 2023) and CREATOR enabling iterative refinement through manual execution (Qian et al., 2023). More recent systems, Alita (Qiu et al., 2025) and ATLASS (Haque et al., 2025), perform need-based tool generation with capability checking against an existing toolset, augmented by web search and code-execution feedback loops. 

## 3 Methods 

El Agente Forjador is a task-driven agent framework in which each task is solved with a toolset that can be generated and modified one the fly to meet the requirement of the task (see Figure 2). Rather than relying on a fixed toolset, the system expands its capabilities on demand. The system operates through a four-stage workflow (tool analysis, tool generation, task execution, and solution evaluation). Supporting this workflow are a tool generator that creates, tests, and iteratively refines reusable Python functions, a toolset optimizer that 4maintains hierarchical organization of the growing toolset, a task executor that carries out script generation, execution, and report compilation, and a solution evaluator that assesses task completion and drives iterative refinement. We implement this workflow with general-purpose coding agents that share a common agent workspace (see Figure 2), enabling failures encountered in one stage to be repaired by later stages without human intervention. We leverage state-of-the-art coding agents, specifically Claude Code (Anthropic, 2025), Codex (OpenAI, 2025), and OpenCode (OpenCode.ai, 2025), as the underlying building blocks of our system. 

3.1 The Forjador workflow 

Each run of El Agente Forjador begins with a task specified in natural language in a question.md file in the agent workspace. When a new task is provided, El Agente Forjador executes a multi-stage workflow in which each stage spawns a fresh coding agent session in the agent workspace with a different prompt. 1. Tool analysis : For each task, the agent surveys the existing toolset in the workspace tools/ directory. It inspects available tools, reviews their source code, and determines whether they meet the task requirements. If gaps exist, the agent drafts a specification for the new tools that are needed. 2. Tool generation : Given the specification of a new tool, the tool generator searches the source code of the scientific software required for the tool, implements the tool in Python, and writes test cases. It then iterates through a review–refinement loop until all tests pass. Details can be found in Section 3.3. 3. Task execution : Once all necessary tools are available, the agent writes task-specific execution scripts that compose the tools from the toolset. Specifically, the agent drafts Python scripts that import the required tools, configure input parameters, and orchestrate the computational pipeline. It then executes these scripts, monitors for errors, and iteratively debugs until the pipeline completes. Finally, the agent compiles the results into a report.md file that addresses the task specification. Details are provided in Section 3.5. 4. Solution evaluation : After execution, an automated evaluator assesses whether the generated report satisfies all task requirements and determines whether additional iterations are needed. If any criterion is not met, the evaluator produces a structured next-step plan for the subsequent iteration. This loop repeats until all criteria are satisfied or the iteration budget is exhausted. The evaluation criteria and mechanism are detailed in Section 3.6. 

Hierarchical Toolset with Progressive Disclosure. Each tool in El Agente Forjador is a Python file stored in a hierarchy of directories (see Figure 3). The hierarchical is built by the toolset optimizer , which we will introduce in Section 3.4. In this way, the agent is not given all existing tools upfront. Instead, it discovers tools through progressive disclosure within this hierarchy. For instance, an agent working on molecular geometry optimization navigates into relevant categories such as molecular_geometry_processing/ 

and hartree_fock_calculations/ , but does not see tools nested under unrelated branches like electronic_-property_analysis/ unless the task requires them. 

Curriculum Learning Through Task Sequencing. The persistent toolset across tasks naturally enables curriculum learning (Bengio et al., 2009; Wang et al., 2023): when tasks are arranged from simple to complex, early agents generate foundational tools that later agents can directly reuse. For example: An early task, such as “optimize the geometry of methane” generates foundational tools for converting molecular strings (e.g., SMILES (Simplified Molecular-Input Line-Entry System) (Weininger, 1988)) to 3D structures and performing geometry optimization using Hartree–Fock (HF). A subsequent task, such as “compute the hydride abstraction energy of ethane”, reuses the structure generation and optimization tools but requires new tools to compute thermochemical properties. A later task such as “compute the pK a of acetic acid in solution” reuses all prior tools and additionally generates tools for implicit solvation and Gibbs free energy corrections. The goal of curriculum learning is to progressively build a toolset that covers the space of tasks in a given domain, so that future tasks can be solved primarily through tool reuse rather than repeated generation. This staged approach ensures that agents build on previously discovered knowledge, amortizing the cost of tool generation across tasks and thereby creating an increasingly capable toolset. As the toolset matures through the curriculum, the proportion of tasks requiring the generation of new tools diminishes. 3.2 Tool Analyzer 

The tool analyzer is responsible for translating a natural-language task into a concrete plan of which tools should be reused and which need to be generated. Given the task description and the current contents of the shared toolset, the analyzer inspects available tools, identifies the computational capabilities required by the task, and produces a structured analysis with three components: (i) a task analysis that decomposes the problem into its computational stages, links each stage to the tool that should perform it, and surfaces relevant domain considerations such as recommended methods, physical formulas, or common pitfalls; (ii) a list of existing tools to reuse , drawn from the current toolset; and (iii) a list of new tool requirements , each specified as an abstract contract — a name, a description of the desired functionality and the underlying package or method to use, and the expected input and output types — without committing to a concrete implementation. 

3.3 Tool Generator 

The tool generator is responsible for creating new tools when the tool analyzer deems existing ones are insufficient for a given task. Each generated tool is implemented as a standalone Python function with strict interface requirements and validation mechanisms. The generator operates in an isolated tool_smith/task_-{id}/ directory within the agent workspace, where it drafts implementations and runs tests without affecting the shared toolset. Once a tool passes all tests, it is copied into the root of the shared tools/ directory, making it available for subsequent tasks. The generator follows an iterative refinement loop: it drafts an initial implementation based on task requirements, generates unit tests, executes these tests, analyzes failures, and regenerates the tool until all tests pass. A separate tool reviewer agent then reviews the tool for correctness, adherence to interface requirements, and alignment with domain-specific generation guidelines. The generator revises the implementation if the reviewer deems it unsatisfactory. 

Local Source Browsing. When generating a tool, the tool generator is prompted to directly browse the 

site-packages directory, where Python packages are installed. We provide coding agents with precomputed navigation indexes that summarize each library’s module structure, import patterns, and common usage, 6enabling the agents to efficiently locate relevant functions without scanning entire packages. This design has two key advantages: (1) it enables tool generation against private or unreleased packages that have no public documentation, and (2) it guarantees that the agent references the exact interface of the installed version, avoiding outdated or version-mismatched information that is common in web-sourced documentation. 

Strict Tool Interface Requirements. To ensure tools are reliable, composable, and debuggable, we enforce several constraints on all generated tools by prompts to the tool generator and tool reviewer: 

• No ad-hoc tools. Every tool must be a general-purpose reusable utility, not a task-specific script. The tool analyzer enforces this via an LLM-based validator that rejects proposals containing one-time scripts unlikely to be reused. This constraint ensures the toolset remains reusable across the curriculum and prevents redundant generation of near-identical single-purpose tools. 

• No silent failures or fallback behaviour. Tools must raise explicit exceptions when encountering errors rather than returning default values, suppressing warnings, or attempting automatic recovery. This strict error propagation is essential for enabling downstream debugging: when a tool fails during execution, the calling agent receives clear diagnostic information and can inspect, modify, and repair the tool. Silent failures would mask bugs and prevent the self-healing behaviour central to our approach. 

• Pydantic validation for type safety. All tool inputs and outputs are validated using Pydantic , a Python library that provides runtime type checking and data validation through declarative schemas. Each tool defines input parameters as BaseModel classes with explicit types, constraints, and documentation. This serves multiple purposes: (1) it enables agents to reason precisely about tool interfaces when composing multi-step workflows, (2) it catches type errors at runtime before they propagate through complex pipelines, and (3) it provides self-documenting code that agents can parse to understand tool capabilities. 

Skill-Based Generation Guidelines. The tool generator can optionally leverage a collection of skills (Anthropic, 2025; K-Dense Inc., 2026)—curated markdown documents that encode domain-specific best practices, common patterns, and expert preferences. When generating tools, agents can read relevant skill documents to inform their implementations. For example, a skill document might direct the agent to use TenPy for density matrix renormalization group ( DMRG ) calculations instead of a from-scratch NumPy implementation, specify the correct pulse optimization interface in QuTiP , or provide a detailed reference for diagnosing imaginary vibrational frequencies in geometry optimizations. This mechanism serves as an important way to compensate for gaps in an LLM’s domain knowledge: rather than relying solely on parametric knowledge from training data, agents can consult authoritative, user-curated guidance at generation time. This allows 

El Agente Forjador to generate tools aligned with established scientific practices. The generation guidelines used in our experiments are provided in Appendix J. 

3.4 Toolset Optimizer 

As agents solve more tasks and the toolset grows, maintaining navigability and context efficiency becomes critical. The tool structure optimizer is a separate coding agent that is triggered at the beginning of each new task, before tool analysis and generation. It recursively scans the tools/ directory and creates new subdirectories to group tools or directories when any directory exceeds a size threshold, ensuring that agents can quickly locate relevant tools without overwhelming their context windows. The optimizer operates through a simple heuristic: when a tool category directory exceeds a threshold number of tools (10 in our configuration), it analyzes tool names, docstrings, and dependency patterns to identify natural subcategories. It then creates new subdirectories, moves tools accordingly, and updates import paths and documentation. Figure 3 illustrates this process: an initially flat toolset is reorganized into subcategories, while newly generated tools are placed at the root awaiting future reorganization. In addition to reorganizing, the optimizer can identify functionally overlapping tools and merge them into unified implementations, thereby preventing unnecessary expansion of the tool library as multiple agents independently generate similar utilities. This feature was not enabled in our benchmark runs, as the toolsets 7remained small enough that merging introduced unnecessary overhead. A full description is provided in Appendix G. 

3.5 Task Executor 

Once all required tools are available, the task executor is invoked to generate and execute the solution pipeline. It receives the original task description, along with the tool analysis plan produced by the tool analyzer, and operates as a general-purpose coding agent within the agent workspace. 

Script Generation and Execution. The coding agent is prompted with a structured instruction that includes the task specification, a navigation index of available Python packages, and the current contents of the 

tools/ directory. Guided by these inputs, the agent designs a multi-phase scripting strategy: it decomposes the computation into logical stages (e.g., preprocessing and input validation, main simulation, and result aggregation), each implemented as a separate Python script. The agent is prompt to first tests its pipeline on a small subset of inputs (two to three molecules or parameter points) to catch errors early, then scales to the full dataset once correctness is verified. Scripts are executed via the Model Context Protocol ( MCP ) execution interface (described below), which dispatches jobs to either a local subprocess or an HPC scheduler depending on the environment. During execution, the agent monitors outputs and logs, iteratively debugging failures until the pipeline completes. Because every agent has full write access to the tools/ directory, it can also debug and modify existing tools on the fly when a reused tool produces incorrect results or raises unexpected errors, enabling self-repair without re-entering the tool generation phase. The agent is prompt to compiles results into a report.md in the workspace that addresses each aspect of the original task specification. The presence and completeness of this report serves as the primary indicator of success for the iteration. 

Model Context Protocol for Command Execution. Scientific computing often requires executing long-running computational jobs on HPC clusters with workload management systems like SLURM (Yoo et al., 2003). To handle this seamlessly, we integrate an MCP server (Anthropic, 2024) that provides agents with robust command-execution capabilities across heterogeneous computational environments. In HPC environments with SLURM , it automatically generates job submission scripts with appropriate resource requests, submits jobs to the scheduler and retrieves results. In local or cloud environments without job schedulers, it executes commands as subprocesses. This abstraction allows agents to focus on scientific logic rather than infrastructure details. 

3.6 Solution Evaluator 

The solution evaluator is a coding agent that determines whether a generated report satisfies the task requirements; if it does not, the task executor is re-invoked with a structured next-step plan to refine the solution, enabling iterative improvement without human intervention. It reads both the task specification (question.md ) and the generated report ( report.md ), and is prompted to browse the workspace to collect additional context to prevent the executor from cheating. It outputs a structured evaluation.json with boolean flags across five dimensions: bug_need_fix (whether bugs are detected that require fixing), script_complete 

(whether all required computations are implemented), further_simulation_needed (whether additional simulation is required), result_complete (whether all required deliverables are present, such as data tables, visualizations, and interpretations), and next_step_needed (whether any of the above conditions are unmet). When further work is required, it also provides an actionable next_step_plan in text, which is incorporated into the updated task specification for the next iteration. This evaluation-refinement loop runs for a maximum of five iterations, after which the task is marked as failed if requirements are still unmet. The detailed prompt can be found in Appendix C. 

## 4 Results 

We evaluate El Agente Forjador across two capability domains: quantum chemistry and quantum dynamics. For quantum chemistry tasks, agents are required to use PySCF (Sun et al., 2017, 2020) as the primary electronic structure backend, supplemented by RDKit (Landrum et al., 2024) and the Atomistic 8Simulation Environment ( ASE ) (Hjorth Larsen et al., 2017). For quantum dynamics tasks, the required software includes CUDA-Q (NVIDIA, 2024) and Qiskit (Javadi-Abhari et al., 2024) for quantum circuit simulation, Tequila (Kottmann et al., 2021) for variational quantum algorithms, QuTiP (Johansson et al., 2012, 2013) for open quantum system dynamics, and TenPy (Hauschild and Pollmann, 2018) for tensor network simulations. 

4.1 Quantum Chemistry 

We first benchmark our system on 13 quantum chemistry tasks, derived from the benchmark set of El Agente Q and El Agente Gráfico , consisting of undergraduate-level tasks, spanning six categories: organic compounds property calculation, inorganic compounds property calculation (both covering geometry optimization, HOMO–LUMO gap, point group symmetry, and Mulliken charges), relative carbocation stability (∆H and ∆G for hydride abstraction), cycloalkane ring strain energies (C 3–C 8), pK a prediction with implicit solvation and proton calibration, and time-dependent density functional theory ( TD-DFT ) absorption spectra (S 1 energy, oscillator strength, S 1–T 1 gap) (Casida, 1995). Most categories are tested at two difficulty levels: Level 1 typically provides simpler or more scaffolded inputs, whereas Level 2 requires agents to handle more complex cases. Additionally, we introduce a Level 0 task for the organic compounds category, a single-molecule analysis of caffeine, to provide a simpler entry point for curriculum learning, allowing the agent to bootstrap foundational tools before tackling the harder tasks (see Table 1). 

4.2 Quantum Dynamics 

The quantum dynamics benchmark comprises 11 tasks drawn from the El Agente Cuántico . These tasks span three categories: (i) quantum circuit implementation and quantum control, including Bell state preparation, variational quantum eigensolver ( VQE ) dissociation curves (Peruzzo et al., 2014; Tilly et al., 2022), noisy channel characterization, and GRAPE-based optimal control of a three-level system (Khaneja et al., 2005); (ii) open quantum system dynamics, including Fenna-Matthews-Olson ( FMO ) exciton dynamics simulated via hierarchical equations of motion ( HEOM ) (Adolphs and Renger, 2006; Tanimura, 2020), transverse-field Ising model magnetization, and Lindblad evolution (Lindblad, 1976); and (iii) many-body simulations, including Floquet kicked Ising dynamics, Floquet dynamics with time-dependent density matrix renormalization group ( TD-DMRG ) (White and Feiguin, 2004; Vidal, 2004), and phase diagram computation via DMRG (White, 1993) (see Table 2). 

4.3 Evaluation 

To evaluate El Agente Forjador , we compare three operating modes across five coding agent setups to show the effect of tool generation and curriculum learning. 

• Zero-shot (ZS): Each problem is solved independently with an empty toolset; no prior tools are available. The four-stage agent workflow is active, so the agent forges an independent toolset for each task, exercising the tool generator and iterative evaluator from scratch. 

• Tool reuse (TR): Each problem is solved with access to a toolset constructed via curriculum learning using Claude Opus 4.6 (via Claude Code) across all problems in the benchmark. The same toolset is then reused by in all the setups. Tool generation is disabled; the tool analyzer remains active, enabling the agent to discover relevant tools through progressive disclosure and compose them. 

• Evaluator only (EO): The agent directly solves the problem without invoking the tool builder or accessing any existing toolset. Solutions are still verified by the automated solution evaluator (Section 3.6). This mode serves as a baseline that reflects the raw capability of the underlying coding agent without tool scaffolding. Each mode–setup combination is repeated 3 times; for tool reuse (TR), each repetition draws from an independently curriculum-learned toolset from Claude Opus 4.6 (via Claude Code) with a fixed order of tasks (see Appendix E.1). Across 24 tasks, 3 operating modes, 5 coding agent setups, and 3 repetitions, the evaluation encompasses 1,080 total task runs. Task success is evaluated against explicit criteria for each task, based on the numerical results and methodology adopted in the solutions. Tool Reuse Consistently Reduces Cost. A central motivation of curriculum learning is to amortize the cost of tool generation across tasks: once foundational tools are forged by an early, stronger agent, later agents skip the expensive tool-generation loop entirely and proceed directly to tool composition. This design yields 11 substantial LLM API cost reductions across both benchmarks and all five models. On quantum dynamics (see Table 12), average cost reductions range from 62.7% for Claude Opus 4.6 via Claude Code to 77.9% for Kimi K2.5 via OpenCode, with a mean reduction of 66.9% across all models. On quantum chemistry (see Table 11), reductions are more modest but still consistent, ranging from 33.2% for Claude Sonnet 4.6 via Claude Code to 64.2% for Kimi K2.5 via OpenCode, with a mean of 49.6%. 4.4 Case Studies 

A key promise of task-driven tool generation is that the resulting toolsets are not confined to the domain in which they were created: tools forged for quantum chemistry should compose seamlessly with tools forged for quantum dynamics, enabling cross-domain workflows. Moreover, when the existing library lacks a required capability, the agent can forge new tools on the fly, extending its repertoire without human intervention. To test this, we present two representative case studies. Both problems require bridging classical electronic-structure theory and open-quantum-system dynamics, a combination that demands tools from multiple domains and, in each case, triggers the generation of new tools to fill gaps in the existing toolset. To set up the case studies, we seeded the agent’s toolset by combining one curriculum-built toolset from the quantum chemistry benchmark with one from the quantum dynamics benchmark, giving the agent a cross-domain toolset to draw from at the outset. Traces of the case studies can be found in Appendix K.

## 5 Discussion 

5.1 Strengths 

Task-Driven Agent Generation Reduces Human Involvement. Our approach significantly reduces the need for human intervention in building agent systems. Instead of relying on manually engineered, domain-specific prompts or hand-crafted tools, agents are generated in a task-driven manner: complex capabilities emerge through task decomposition and tool generation. This paradigm aligns well with the strengths of LLMs, whose training data already encode most of the relevant procedural and domain knowledge. In practice, describing a target task or simulation objective is often easier than designing expert-level prompts. 

Tool Reuse Simultaneously Improves Cost, Speed, and Accuracy. In most machine learning settings, improve-ments in one metric come at the expense of another. Tool reuse breaks this trade-off: across five coding 16 agent setups, we observe simultaneous reductions in API cost (33–78%) and wall-clock time (up to 88%), with equal or improved task accuracy. This is because curriculum-built tools amortize the exploration cost of tool generation: once a useful tool has been discovered and validated, it can be reused at negligible marginal cost. The result is a progressively more efficient system where per-task cost decreases as the toolset grows. Moreover, because the curriculum toolset is built by a stronger model (Claude Opus 4.6) and reused by weaker ones, tool reuse also serves as strong-to-weak knowledge transfer: correct domain methodology is encoded in the tools and can be used by weaker agents. 

Universal Agents Enable Self-Healing Tool Ecosystems. Rather than assigning agents fixed context and roles, all agents in our framework are universal: any agent can code, debug, and browse the whole agent workspace. This design creates a self-healing ecosystem. Tools are often imperfect at the time of initial generation but are subsequently corrected by downstream agents. Because later agents depend on these tools to solve their own tasks, failures naturally trigger inspection, refinement, and repair, all without explicit human supervision. In practice, bugs surface when tools are applied in new contexts, and the same agent that encounters the bug has the capability to fix it—an effect quantified by the tool-editing statistics in Appendix H. The curriculum ordering amplifies this effect, as tools generated for simpler tasks are tested by increasingly complex downstream applications. 

5.2 Limitations & Outlook 

We identify several limitations of the current system and outline directions for future work. 

Unreliable Self-Verification and Reward Exploitation. Our system relies on agent self-verification and au-tomated evaluation signals. As with many autonomous agent frameworks (Hu et al., 2024; Zhang et al., 2025), these signals are not always reliable. Agents may incorrectly assess correctness, fail to detect subtle errors, or converge to behaviours that exploit weaknesses in the evaluation function, optimizing for superficial success metrics rather than true task correctness. This risk is particularly acute in scientific computing, where numerically plausible but physically incorrect results (e.g., a geometry optimization converging to a saddle point rather than a minimum) may pass automated checks. A promising mitigation is evaluation-time scaling: allocating more computation to verification and refinement, analogous to test-time compute scaling in language models (Snell et al., 2024). Recent work has taken important steps toward verifiable and self-correcting AI agents for quantum many-body simulations (Deng et al., 2026). Domain-specific invariants (conservation laws, symmetry constraints, known asymptotic limits) could serve as physics-informed verification signals that are harder for agents to exploit than generic correctness metrics. 

Restricted Software Ecosystem. The current implementation is limited to Python-based tools and a predefined set of repositories. While this simplifies execution and reuse, it excludes tasks that require input-file-driven software, such as ORCA (Neese et al., 2020), VASP (Kresse and Furthmüller, 1996), or Gaussian (Frisch et al., 2016), external APIs, or non-Python ecosystems, which remain prevalent in computational chemistry. Extending the framework to wrap such software (generating input files, launching executables, and parsing output) is a natural next step that would substantially broaden the system’s applicability without abandoning the Python-based tool abstraction layer. 

Toward Self-Directed Curriculum Generation. The current system requires human-specified task sequences to drive curriculum learning. Future versions of the system could integrate self-tasking mechanisms (Hu et al., 2025; Xia et al., 2025; Zhai et al., 2025), in which the agent identifies gaps in its toolset and generates synthetic problems to address them, analogous to how Voyager (Wang et al., 2023) grows skills through self-directed exploration. Combined with automated verification, this would close the loop from tool generation to validation without human intervention, moving toward a fully autonomous scientific agent that expands its own capabilities over time. 17 6 Conclusion 

We present El Agente Forjador , a multi-agent framework that shifts the bottleneck in scientific automation from building tools to specifying tasks . The system autonomously forges, validates, and composes reusable computational tools through a four-stage workflow driven by task requirements rather than manual engineering. Across 24 tasks spanning quantum chemistry and quantum dynamics on five coding-agent backends, curriculum-based tool reuse cuts API cost by 33–78% and wall-clock time by up to 88%—without sacrificing accuracy. Moreover, encoding domain methodology into reusable tools raises average scores from 81.5% to 85.8% on quantum chemistry and from 91.5% to 93.2% on quantum dynamics, with weaker models gaining as much as 16.5 percentage points—approaching the performance of frontier models at a fraction of the cost. Case studies further show that tools forged in separate domains compose to solve hybrid problems, with any missing capability forged on demand. These results establish a paradigm in which scientific agent systems are defined entirely by the tasks to be solved: researchers specify what to compute, and the computational infrastructure for doing so emerges and self-improves through use.

## F Benchmark Evaluation Procedure 

Each agent run is evaluated along two independent axes, accuracy and methodology , using automated LLM-as-judge pipelines. The procedure is identical across both benchmark suites with minor adaptations described below. 49 Accuracy evaluation. For the quantum chemistry tasks, the evaluator first extracts numerical results from the agent’s output into a structured JSON file, which is then scored by a deterministic Python script that compares values against ground-truth references with appropriate tolerances. For the quantum dynamics tasks, a separate Claude Code instance (with Claude Sonnet 4.6) reads the task specification and the agent’s output files, then judges each predefined criterion as true or false . The accuracy score is the fraction of satisfied criteria. 

Methodology evaluation. For both benchmarks, each task is accompanied by a set of weighted methodology stages that describe the expected workflow (e.g. Hamiltonian construction, solver selection, post-processing). A Claude Code-based evaluator (with Claude Sonnet 4.6) reads the agent’s workspace and scores each stage on a 0–10 scale based on whether the agent followed correct domain methodology, independent of numerical accuracy. The final methodology score is the weighted average across all stages. Valid alternative approaches that achieve the same scientific objective receive full credit. 

## G Tool Merger 

Over time, given enough different tasks, El Agente Forjador ends up creating sets of tools that are similar or identical in functionality. These excess tools negatively affect navigability and context efficiency. Therefore, we designed a tool merger , a part of the toolset optimizer that merges and deletes existing tools to reduce clutter. Tool merging follows a simple and efficient pipeline. Docstrings and function signatures of all tools are extracted and embedded before being sorted into clusters via embedding similarity. Identified clusters are then passed to Tool Merger, which decides if and how they should be merged. Tools that require merging are stored in a temporary backup directory, and new merged tools are drafted. These are then passed to the tool verification agent to ensure their correctness before deleting the original tools. If a merged tool fails verification, feedback is passed back to the tool merging agent along with the original tools to try again. Upon any re-failure, the original set is restored to the main directory. The tool structure optimizer is called following any tool merging to ensure optimal organization. In this way the smallest fully functional tool set possible can be maintained to ensure optimal navigability and context efficiency. Here we present and demonstration run in which the tool merger works on a set of tools created by solving the quantum chemistry tasks. Tool merger identified two similarity clusters containing six tools in total. The first cluster grouped four DFT tools into two merge pairs: the gas-phase and solvation variants of both single-point energy calculations and geometry optimizations. In each pair, neither tool was a strict superset of the other: the solvation variants added implicit solvent support (CPCM/DDCOSMO) but lacked features such as density fitting and auxiliary basis selection present in the base tools, while the geometry optimization pair additionally suffered from inconsistent input and output field names ( spin_multiplicity vs. spin , energy 

vs. final_energy ). Tool merger unified each pair into a single tool that makes solvation an optional parameter while standardizing all interfaces. The second cluster paired a thermochemistry tool with a vibrational frequency tool. Both internally computed the molecular Hessian via the same PySCF call, but the thermochemistry tool returned only thermodynamic quantities (ZPE, enthalpy, Gibbs free energy, entropy) filtered to real positive frequencies, while the vibrational tool returned the full frequency spectrum including imaginary modes and normal-mode displacement vectors. The unified replacement computes the Hessian once and returns both the complete vibrational analysis and all thermodynamic properties, eliminating a redundant and expensive matrix computation. All three unified tools passed automated review by a separate verification agent on the first attempt with zero errors or retries, reducing the tool inventory from 18 to 15. The remaining 12 tools were found to be sufficiently distinct from one another and were left unchanged. Original source files were preserved in a backup directory to allow rollback if needed. 50 H Tool File Edits During Execution 

El Agente Forjador ’s iterative problem-solving loop gives the coding agent full write access to tool source files, allowing it to revise tools on the fly when execution results suggest a bug or a needed improvement. We expected this self-repair mechanism to be exercised frequently, yet our experiments show that in-place tool edits during execution were surprisingly rare for most setups. Table 18 reports how often tools were modified during execution. We distinguish two kinds of modification: 

pre-generated tool edits , where the agent rewrites an existing tool file, and new tool creation , where the agent writes an entirely new tool file that did not previously exist. In the zero-shot setting, Kimi K2.5 (via OpenCode) created new tools in 29.9% of its 144 iterations and edited pre-existing tools in 13.9%, while Gemini 3.1 Pro (via OpenCode) created new tools in 30.6% of 98 iterations and edited pre-existing ones in 12.2%. GPT-5.2-Codex (via Codex) edited pre-existing tools in 12.8% of iterations and created new ones in 4.5%. By contrast, Opus 4.6 (via Claude Code) edited tools in only 6.0% of iterations and Sonnet 4.6 in 1.2%, with neither model ever creating new tools during execution. In the tool reuse setting, only Gemini 3.1 Pro and GPT-5.2-Codex edited tools by 3.9% and 3.3% of iterations, respectively; every other configuration left its tools untouched. These results suggest that the quality of initially generated tools differs greatly across models: stronger models produce tools that work correctly on first use, while weaker models must fall back on runtime patching, or even ad-hoc tool creation, to compensate for deficiencies in their initial implementations. The near-absence of edits in the tool reuse setting reinforces this interpretation, as the pre-existing library was generated by a stronger model (Opus 4.6) and thus requires little correction regardless of which model consumes it.

## J Tool Generation Guidelines 

The following are example skill documents (generation guidelines) provided to the tool generator agent during our experiments. Each guideline encodes domain-specific best practices for a particular computational task. For El Cuántico tasks, the LLM often possesses broad knowledge of quantum computing concepts but tends to implement everything from scratch using NumPy rather than leveraging established domain-specific libraries. The DMRG and pulse optimization guidelines therefore steer the agent toward using dedicated packages ( TenPy and QuTiP , respectively) that provide optimized, well-tested implementations. For quantum chemistry tasks, the agent generally knows which computational chemistry packages to use but sometimes lacks the domain knowledge needed to diagnose and resolve specific numerical issues. The vibrational frequency guideline, for example, provides a detailed reference on how to properly analyze imaginary frequencies and verify projection quality, knowledge that is critical for obtaining physically meaningful results but is not always reliably captured in the LLM’s parametric knowledge.

> Prompt Theory overview (what is being projected and why)

For a molecule with N atoms: 

• Total Cartesian degrees of freedom: 3N

• Translational modes: 3

• Rotational modes: 

– 3 for non-linear molecules 

– 2 for linear molecules 

• Vibrational modes: 

– 3N − 6 (non-linear) 

– 3N − 5 (linear) 

> Vibrational frequencies come from

F = M−1/2HM −1/2

where: 

• H: Cartesian Hessian (second derivatives of energy) 

• M: diagonal mass matrix 

• F: mass-weighted Hessian The translation and rotation subspace must be projected out before diagonalization, otherwise you will see near-zero or imaginary frequencies. PySCF provides this projection automatically if you use its harmonic analysis utilities correctly. 

> Hessian calculation

Choose the Hessian module consistent with your SCF type.

> O 0 0 0 H 0 -0.757 0.587 H 0 0.757 0.587
> ''' ,basis = 'cc-pvdz '
> )mf = scf.RHF(mol).run() hess = rhf.Hessian(mf).kernel()

53 res = thermo.harmonic_analysis(mol, hess) print(res[ 'freq_wavenumber ']) 

• hess has shape (3N, 3N) 

• Units: Hartree / Bohr 2

> Verifying projection quality (best practice)

Check that: 

• No frequencies below ∼10 cm −1 (except numerical noise) 

• Number of modes equals: 

– 3N - 6 (non-linear) 

– 3N - 5 (linear)

PySCF automatically detects linearity from geometry and handles: 

• 5 projected modes for linear molecules 

• 6 projected modes for non-linear molecules No manual intervention needed unless geometry is ill-conditioned.

Tool Analysis Report 

> Task Analysis

The task requires a multi-step workflow for ethylene: 1. Geometry generation & optimization : Use smiles_to_3d_geometry to convert ‘C=C’ SMILES to initial 54 3D coordinates, then use run_dft_geometry_optimization with B3LYP/def2-SVP to get the optimized geometry. 2. TD-DFT excited state : Use run_tddft_calculation on the converged DFT mean-field object to compute the S 1 excitation energy ( π → π∗ transition), getting excitation energies and oscillator strengths. 3. MO analysis : Use analyze_molecular_orbitals to identify the HOMO ( π) and LUMO ( π∗) orbital indices and energies. Use the new visualize_molecular_orbitals tool to create contour plots of the π
