## ABSTRACT 

Agentic AI systems are becoming commonplace in domains that re-quire long-lived, stateful decision-making in continuously evolving conditions. As such, correctness depends not only on the output of individual model calls, but also on how to best adapt when incorpo-rating new evidence or revising prior conclusions. However, exist-ing frameworks rely on imperative control loops, ephemeral mem-ory, and prompt-embedded logic, making agent behavior opaque, brittle, and difficult to verify. This paper introduces Credo , which represents semantic state as beliefs and regulates behavior using declarative policies defined over these beliefs. This design supports adaptive, auditable, and composable execution through a database-backed semantic control plane. We showcase these concepts in a decision-control scenario, where beliefs and policies declaratively guide critical execution choices (e.g., model selection, retrieval, corrective re-execution), enabling dynamic behavior without requiring any changes to the underlying pipeline code. 

## 1 INTRODUCTION 

Imagine a financial analyst asking an LLM pipeline: “What was the year-over-year revenue change for Apple in Q3 2023?” The pipeline has to decide, upfront, how extensively to retrieve evidence, which model to run, and how strictly to validate the answer, all without knowing yet whether the query is just a straightforward lookup or requires a multi-document computation. Most LLM orchestration frameworks [ 1 , 10 ] fix these decisions at development time by hardcoding parameters like retrieval depth, model selection, and verification strategy in imperative control flow or fixed prompt templates. A pipeline optimized for simple queries is efficient on easy questions but quietly fails to handle harder ones, whereas a pipeline optimized for complex questions will overspend resources on every request. Both approaches are suboptimal, and neither can be improved without changing the application itself. To illustrate this, we ran 100 FinanceBench [ 4] tasks across all combinations of three model tiers, three prompting strategies, and two retrieval methods, yielding 18 fixed configuration pipelines. Figure 1 shows the results, which highlight two key findings. First, retrieval choice dominates model size. Fixing the model (12B param-eters) and strategy (DSPy [ 5] chain-of-thought module), switching retrieval from dense similarity search to BM25 keyword search drops accuracy from 72% to 45%, a 27-point swing, at nearly identi-cal inference cost. More strikingly, the strongest model (27B) with BM25 retrieval achieves only 50% accuracy, below the weakest model (4B) with dense retrieval at 65%, despite costing more. Sec-ond, prompting strategies can simultaneously improve accuracy and reduce costs. A pipeline that adopts a single configuration for all tasks inevitably either reserves excess capacity it never actually uses or breaks down on tasks that need more resources. This trade-off motivates a system that monitors task characteristics and evidence, and then selects an appropriate configuration on a per-task basis. Two families of systems address related problems but leave a critical gap. Pipeline processing systems such as LOTUS [ 8 ] and DocETL [ 9] expose LLM-powered operators (e.g., semantic joins, filters, aggregations) as composable, SQL-like abstractions. Their strength is declarativity, meaning that developers specify what to compute while the system handles physical execution. However, these physical plans are fixed at compile time and typically do not adapt as semantic signals emerge from intermediate results. At the other extreme, autonomous agent frameworks such as ReAct [ 12 ]and AutoGen [ 10 ] allow fully dynamic, multi-step decision-making driven entirely by LLM reasoning. Their strength is adaptivity because the agent reevaluates its plan at every step, but adaptation logic is embedded in prompt reasoning, making it difficult to inspect, audit, tune, or enforce correctness over. Credo targets the gap between these two families by offering pipelines that are declarative like LOTUS but adaptive like ReAct, with all control decisions explicit, inspectable, and stored as queryable database state.

In Credo , developers specify a logical pipeline while a database-backed control plane governs physical execution. Credo represents runtime semantic state as typed beliefs (e.g., query complexity, evi-dence relevance), expresses adaptation logic as declarative policies 

over those beliefs, and uses a reactive execution engine to evaluate policies at both compile time and runtime. As new evidence arrives, beliefs are reevaluated, policies are triggered, and the pipeline may be rewritten accordingly. Every execution decision is therefore ex-plicit, traceable, and tunable without modifying application code. In this demo, audience members will have the opportunity to interact with Credo and observe these distinguishing features firsthand in the context of the FinanceBench [4] QA benchmark. 

## 2 CREDO 

Credo is built around the idea that logical definition of a pipeline (i.e., which operators run and in what order) should be independent of physical execution decisions (e.g., which model to use, how much evidence to retrieve, how to verify a result) [ 2]. Credo achieves this separation through three complementary layers (Figure 2): (1) an observation layer that makes semantic runtime state explicit as beliefs; (2) a decision layer that translates that state into declar-ative control through policies; and (3) an execution layer that applies those decisions reactively as the pipeline runs. Critically, be-liefs, policies, and the full execution context are stored persistently in a DBMS, making every control decision a queryable, auditable database record rather than a temporary in-process state. 

## 2.1 Observation Layer: Beliefs 

The first challenge in building an adaptive pipeline is knowing what to adapt to. In existing systems, contextual signals such as query difficulty, evidence quality, and model confidence either go unobserved or are consumed implicitly inside prompt logic with no persistent record. Credo introduces beliefs : typed, persistent se-mantic claims over the execution context (e.g., queries, intermediate results), each materialized as a (value , confidence ) pair capturing the system’s current (potentially uncertain) interpretation of prop-erties like query complexity and evidence relevance. Beliefs are computed by extractors (e.g., heuristics, classifiers, LLM prompts) operating on direct evidence or other beliefs. Static beliefs are re-solved entirely before execution begins, whereas dynamic beliefs 

are extracted from intermediate operator outputs at runtime. Figure 3 shows two belief examples. The query_complexity be-lief wraps RouteLLM [ 7 ], a lightweight matrix-factorization router that scores how likely a query is to benefit from a stronger model. Because it operates purely on the input query and requires only an embedding call, it is a static belief extracted once at compile time, before any retrieval or generation occurs, and its score im-mediately informs downstream policy decisions such as retrieval depth and model selection. The relevance belief, by contrast, is dynamic. It is an LLM-as-judge extractor that evaluates whether the retrieved evidence actually contains the information needed to answer the question, returning a (value , confidence ) pair, where value ∈ {correct , ambiguous , incorrect }. Because it depends on the evidence produced by the retrieve operator, it cannot be re-solved until retrieval completes, and its value may trigger corrective rewrites that re-execute upstream steps. The trade-off is extrac-tion cost, necessitating running an expensive LLM-based extractor. 

Credo mitigates this by tracking cost per belief and evaluating cheaper extractors first, only escalating when a decision requires a higher-fidelity signal. All beliefs expose both a value and a con-fidence score, enabling policies to reason not only about system state but also the reliability of that state. 

## 2.2 Decision Layer: Policies 

Knowing the state of execution is necessary but not sufficient. Exist-ing frameworks that do expose runtime signals, such as CRAG [ 11 ]for retrieval quality or self-refinement [ 6 ] for answer quality, hard-code the response to those signals, meaning the adaptation behavior is fixed and cannot be tuned without modifying the application. 

Credo externalizes this logic into policies : declarative rules stored in a catalog that map belief conditions to physical operator param-eters. A policy specifies a target operator, a condition over belief attributes (e.g., value and confidence), and a set of physical parame-ters to apply. Figure 3 shows two examples. The first is an operator rewrite that fires when query_complexity.value ≥ 0.25 and its confidence is sufficiently high, rewriting the generate operator to use the strongest available model. Queries that score below this threshold continue to use a cheaper model, paying for a stronger one only when complexity warrants it. The second is a pipeline rewrite that fires when relevance.value = “incorrect” , in-dicating that the retrieved evidence cannot support an answer to the question. The policy declares an INVALIDATE set covering 

{evidence, model_answer, verified} and rewrites retrieve 

to use a dense embedding index with more candidates. Rather than a simple local parameter change, the reactive engine discards the stale context keys and re-executes all affected operators in the next cycle, structurally adapting the pipeline’s execution path at runtime. Much like a database administrator might declare an index to optimize access paths without altering a SQL query, Credo allows users to author policies independently of pipeline logic, and poli-cies can be modified or extended at any time without touching application code. This design offers complete transparency, since every execution decision can be traced back to a specific policy and the belief values. 

## 2.3 Execution Layer: Actions 

The execution layer is where physical work happens. Credo pro-vides a set of built-in operators (Retrieve , Generate , and Verify )that handle the heavy lifting of each pipeline stage, namely docu-ment retrieval over vector or keyword indices, answer generation via configurable LLM prompting strategies, and verification through methods such as LLM-as-judge and chain-of-verification [ 3]. Op-erators are also extensible, such that developers can register user-defined operators to incorporate arbitrary computation into the pipeline (e.g., external API calls, web search, SQL execution over structured data, domain-specific tools). In the future, we plan to extend Credo ’s built-in operators to support richer tool categories, including multi-modal processing (e.g., chart or table understand-ing), code interpreters, and memory retrieval over long-term agent state. Recall that Credo ’s key design decision is the separation of what an operator does from how it is executed. Operators have no direct knowledge of beliefs or policies; they simply receive parameters and execute. Rather, all orchestration decisions are the responsibility of the control plane, choosing which operator runs next, with what parameters, and whether a prior operator must rerun. This mirrors the separation between a relational operator (e.g., a join) and the query optimizer that decides when and how to deploy it (e.g., hash vs. sort-merge). Credo generalizes this principle to LLM pipelines. The policy layer issues operator rewrites that bind parameters to an operator before it runs (e.g., routing to a stronger model, increasing retrieval depth), and pipeline rewrites that invalidate prior outputs and reschedule upstream operators when runtime beliefs demand correction. This yields a system in which operators can be built in isolation, while the pipeline’s overall adaptive behavior is fully governed by the declarative policy catalog. 

## 3 DEMO DESCRIPTION 

The demo centers around FinanceBench [ 4], a realistic financial QA benchmark whose questions span simple numerical lookups, multi-step calculations, and cross-document reasoning over raw PDFs of financial documents. This domain exercises the full breadth of Credo , as different questions require different retrieval depths, generation strategies, and verification rigor, making it a natural testbed for belief-driven policy rewriting. The audience will interact with the Credo web interface (Figure 4), running live examples against a user-defined pipeline, belief, and policy set. As a jumping-off point, we will begin with two questions drawn from FinanceBench that appear superficially similar but demand very different physical execution plans. The first, “What was Apple’s total revenue in FY2022?” , is a direct numerical lookup that requires locating a single value appears verbatim in the income statement. The second, “What is Microsoft’s FY2022 gross profit margin?” , re-quires locating two figures from the same table (revenue and cost of revenue), then performing a two-step computation. A static pipeline would use the same model and retrieval depth for both, wasting resources on the first and risking errors on the second. The demo will illustrate how users can solve this problem us-ing Credo to specify beliefs and policies. Before retrieval, we reg-ister the RouteLLM router [ 7 ] as a belief, which we showed in Figure 3, and select a retrieval depth and generation model via operator-rewrite policies, such as a score below 0.12 uses BM25 with 𝑘 =6 and the 4B model. After retrieval, a CRAG-style relevance belief evaluates whether the retrieved evidence actually answers the question. If it returns incorrect , a pipeline-rewrite policy invali-dates the evidence and triggers a corrective re-retrieval pass with a broader dense index. After generation, a complex_calculation 

belief inspects the question and the model’s answer to determine whether the answer required combining data across multiple fi-nancial tables or executing more than a single formula. When 

complex_calculation = True , a corrective policy invalidates the draft answer and re-generates using the 27B model. Finally, two free beliefs, groundedness and has_hallucinations , are taken directly from the verifier. If it assigns groundedness ≤ 2 and de-tects hallucinations, a second corrective policy uses its critique to regenerate the response. The audience will use the Credo web interface (Figure 4) to issue queries from the FinanceBench benchmark, or custom queries of their own. ## 4 CONCLUSIONS & FUTURE WORK 

In this demo, we showed how Credo manages LLM pipelines as adaptive plans through declarative beliefs and policies. Moving for-ward, we aim to develop richer declarative APIs for complex agentic workflows and explore learned belief discovery to automate policy authoring. Furthermore, we plan to deepen DBMS integration to better optimize long-lived, stateful decision-making.
