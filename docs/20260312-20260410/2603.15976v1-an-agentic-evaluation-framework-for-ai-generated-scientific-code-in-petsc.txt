# An Agentic Evaluation Framework for AI-Generated Scientific Code in PETSc 

Hong Zhang 1, Barry Smith 1, Satish Balay 1, Le Chen 1, Murat Keceli 1,Lois Curfman McInnes 1, and Junchao Zhang 1

Argonne National Laboratory, Lemont, IL 60439, USA 

hongzhang@anl.gov, bsmith@petsc.dev, balay@anl.gov, lechen@anl.gov, keceli@anl.gov, mcinnes@anl.gov, jzhang@anl.gov 

Abstract. While large language models have significantly accelerated scientific code generation, comprehensively evaluating the generated code remains a major challenge. Traditional benchmarks reduce evaluation to test-case matching, an approach insufficient for library code in HPC where solver selection, API conventions, memory management, and per-formance are just as critical as functional correctness. To address this gap, we introduce petscagent-bench , an agentic framework built on an 

agents-evaluating-agents paradigm. Instead of relying on static scripts, 

petscagent-bench deploys a tool-augmented evaluator agent that com-piles, executes, and measures code produced by a separate model-under-test agent, orchestrating a 14-evaluator pipeline across five scoring cat-egories: correctness, performance, code quality, algorithmic appropriate-ness, and library-specific conventions. Because the agents communicate through standardized protocols (A2A and MCP), the framework enables black-box evaluation of any coding agent without requiring access to its source code. We demonstrate the framework on a benchmark suite of re-alistic problems using the PETSc library for HPC. Our empirical analysis of frontier models reveals that while current models generate readable, well-structured code, they consistently struggle with library-specific con-ventions that traditional pass/fail metrics completely miss. 

Keywords: Agentic evaluation · agents-evaluating-agents · AI code gen-eration · PETSc 

## 1 Introduction 

Large language models (LLMs) are increasingly being applied to scientific com-puting tasks, including code generation [7,25], code translation [6], and simula-tion workflow automation [31]. Yet production-level scientific code is rarely writ-ten from scratch. Through decades of research and engineering practice, scalable HPC libraries such as PETSc [3], OpenFOAM [16], and others, have been devel-oped. Solvers, data structures, and conventions in these libraries encode wisdom about numerical stability, parallel scalability, and resource management. The practical skill an LLM must demonstrate in scientific computing is not imple-menting a conjugate-gradient solver or a Runge–Kutta integrator from scratch,

Existing code-generation benchmarks (Sect. 2) largely ignore this skill. They usually use LLMs to generate standalone functions and evaluate them through test-case matching, reducing assessment to a binary pass/fail outcome or a pass@k metric. HPC-oriented and domain-specific benchmarks typically check whether the generated code compiles and produces correct output. But for library-based HPC code, passing tests is necessary but not sufficient : production quality requires dimensions beyond pass/fail benchmarks: appropriate solver se-lection, adherence to strict API conventions, proper error handling, and efficient use of the library’s abstractions. Evaluating these dimensions requires more than static test scripts. We ad-vocate for moving from scripts evaluating code to agents evaluating agents : a tool-augmented evaluator agent that can compile code against a library instal-lation, execute it in a parallel environment, measure runtime performance, and apply both deterministic checks and LLM-based quality assessments. We real-ize this idea in petscagent-bench (Fig. 1), choosing PETSc (with over 6,000 functions spanning time integrators, nonlinear solvers, linear solvers, data man-agement, and optimization) as the test case, because its breadth and complexity make the benchmark challenging and realistic. 1

## 2 Related Work 

Recent advances in LLMs have begun transforming scientific software engi-neering. These models have demonstrated strong capabilities in tasks such as code generation [5,26,29], HPC software development [14], and compiler valida-tion [23]. Consequently, a growing body of benchmarks has emerged to evaluate LLM performance on programming and scientific computing tasks. 

Code Generation Benchmarks. HumanEval [7] and MBPP [2] evaluate LLM-generated Python functions using unit tests, while SWE-bench [19] and CodeContests [20] extend evaluation to repository-level tasks and algorithmic reasoning. ScienceAgentBench [8] and AInsteinBench [12] further benchmark agents operating on scientific codebases. However, these benchmarks operate primarily in Python and focus on functional correctness without evaluating com-pilation, parallel execution, or numerical accuracy. 

HPC and Scientific Code Generation. ParEval [25] benchmarks paral-lel code generation across six programming models and has been extended to repository-level tasks [10]. HPC-Coder-V2 [4] fine-tunes LLMs for parallel pro-gramming languages, while ChatHPC [30] proposes an AI-assisted HPC ecosys-tem with trustworthiness guarantees. Other work evaluates models such as DeepSeek [24] and studies performance optimization [9] and legacy code trans-lation [17,11]. Domain-specific benchmarks include FEM-Bench [22] for finite elements, CFDLLMBench [28] for computational fluid dynamics, and SciML Agents [13] for ODE solver generation. Even when these benchmarks measure performance or numerical accuracy, they do not assess solver selection, API conventions, or library best practices. petscagent-bench addresses this gap by targeting library-based code generation with a multi-dimensional evaluation pipeline, testing whether LLMs can correctly use PETSc solvers, data structures, and conventions. 

Agent Architectures and Evaluation. LLM-as-a-judge [32] uses a language model to score text outputs in a single turn, but the judge has no tools: it cannot compile code, run binaries, or measure execution time. Multi-agent frameworks such as MOSAIC [27], PDE-Agent [21], and MooseAgent [31] use agents for code 4 H. Zhang et al. 

generation but evaluate with traditional test suites. petscagent-bench goes further: its evaluator is itself a tool-augmented agent that compiles, executes, measures, and judges code through a multi-stage pipeline, communicating with the model under test via standardized protocols (A2A and MCP), an agents-evaluating-agents paradigm that, to our knowledge, has not been applied to scientific code-generation benchmarking. 

## 3 An agentic evaluation framework for HPC libraries 

3.1 Design Principles 

Traditional code-generation benchmarks follow a simple pipeline: present a prompt, collect the generated code, and run it against test cases. This approach works well for self-contained Python functions but breaks down for scientific HPC code that must be compiled against external libraries, linked with MPI, executed in a parallel environment, and assessed along dimensions beyond functional cor-rectness. A significant engineering challenge is that the evaluation infrastructure itself must manage library installations, handle compilation failures gracefully, and orchestrate multiple evaluation steps. An agentified architecture addresses this challenge by decoupling the model under test and the evaluator into independent agents and making the evalua-tor an autonomous agent rather than a static test harness. In particular, our 

agents-evaluating-agents architecture stands on three principles: (1) role-based separation, where the evaluator and the model under test are independent agents that interact solely through A2A, and any agent speaking this protocol can be easily plugged into the evaluation infrastructure; (2) sandboxed execution, where generated code is compiled and run in a sandboxed PETSc/MPI environment through MCP tools; and (3) modular extensibility, where adding a benchmark problem requires only a JSON specification and adding an evaluator requires implementing a single Python class. 

3.2 Component Overview 

The system consists of three major components as shown in Fig. 1: 

– Green agent (evaluator). The Green agent is an assessment manager that orchestrates the benchmark. It loads problem specifications from JSON files, dispatches each problem to a Purple Agent through A2A messages, receives generated code and auxiliary information, invokes MCP tools for compila-tion and execution, runs the 14-evaluator pipeline (see Sec. 4), and prepares structured reports for a web-based leaderboard. 2

– Purple agent (model under test). The Purple agent is an LLM-based code-generation agent that receives a natural-language problem description through A2A and returns one or more C source files together with metadata including the entry-point file, dependencies, and command-line arguments.

– MCP server (tool provider). The server provides a sandboxed execution environment exposing compilation and execution tools to the Green agent through MCP. It manages a custom PETSc installation, updates build recipes, compiles uploaded source files, runs the resulting binaries with MPI, and re-turns outputs (including errors and exit codes). 

3.3 Protocol Usage 

The interaction between the Green agent and Purple agent is governed by the A2A protocol [15]. The Green agent creates a task containing the problem de-scription and the Purple Agent responds with artifacts , such as source files and metadata. This decoupled setup allows the two agents to run on different ma-chines, to sit behind firewalls, or be hosted as cloud services. The MCP [1] protocol governs the Green agent’s access to the compilation and execution environment. The MCP Server exposes two primary tools: (1) 

compile , which builds the uploaded code against PETSc, and (2) run , which executes the binary with specified MPI ranks and command-line arguments. It is deployed for free public access. See https://mcp.petsc-ai.org for details. 

## 4 Evaluation Pipeline 

The evaluation pipeline contains three sequential stages: gates , metrics , and 

quality assessments . If any gate fails, evaluation terminates immediately with a zero score, so only compilable and executable code proceeds to the later stages, which involve more expensive analysis. Independent of the stages, each evalua-tor is attributed to one of five scoring categories that group similar dimensions for aggregation: correctness (syntactic, functional, semantic, and numerical), 

performance (execution time), code (readability, style, documentation), ap-propriateness (algorithm and solver selection), and library-specific (API con-ventions, error handling, parallel awareness). Table 1 summarizes all evaluators. 

4.1 Stage 1: Gates 

Gates act as hard filters: evaluation continues only if the generated code passes all the following gates: 

– Compilation . The C/CUDA/Kokkos source files compile against PETSc us-ing PETSc’s Makefile system. 

– Execution . The compiled binary runs to completion without crashes or er-rors. 

– Memory safety. The submitted code must have no leaks or invalid memory accesses. This gate leverages runtime analysis tools (e.g., Valgrind) or PETSc’s own memory tracking features to detect memory issues. 

– API usage. The code must follow correct PETSc patterns (e.g., bracketing of PetscInitialize /PetscFinalize , using PETSc public header files). 6 H. Zhang et al.

PETSc’s source repository is public and almost certainly part of LLM train-ing data. To mitigate data contamination [18], we intentionally alter initial conditions, boundary conditions, or discretizations if similar problems exist in PETSc tutorials. By doing so, we ensure that models cannot simply reproduce memorized solutions, thereby increasing the reliability of the benchmark as a test of generalization rather than memorization. For example, PETSc’s tutorial suite solves Darcy flow with finite differences; our specification requires a finite-element formulation instead, forcing the model to use higher-level abstractions such as PETSc’s DMPLEX module for mesh management and element assembly. 

## 6 Evaluation Results and Analysis 

This section presents a rigorous, reproducible evaluation of agent performance on our benchmark. We analyze how different models perform across the problems, categories, and failure modes. The results reveal category-level strengths and weaknesses and offer practical guidance for improvement. 

6.1 Evaluation Settings 

Each agent uses a frontier LLM (Claude Opus 4.6, Gemini 2.5 Pro, or GPT-5.2) and is given only one shot to generate code to solve each problem. Each test is repeated three times under identical conditions to account for variability in LLM outputs. The Green agent uses GPT-5.2 for LLM-based quality evaluators since it is known for its versatility. All models are used with temperature set to 0 to facilitate reproducibility. All problem specifications, evaluation code, and configuration files are publicly available at https://github.com/petsc/ petscagent-bench .

6.2 Overall Performance 

All models achieve only modest average composite scores. Gemini 2.5 Pro leads at 46.4 , followed by Claude Opus 4.6 at 42.4 and GPT-5.2 at 39.9 . Across 54 total attempts, 21 score zero due to gate failures, while 17 exceed 80. petscagent-bench 9

6.3 Per-Problem Analysis 

A clear difficulty gradient can be seen in Figure 2. The easy problems, Rosen-brock and Vec/MPI Scatter, score 82–93 when gates are passed, although Claude fails all three Vec/MPI attempts. Medium problems exhibit greater variability: 1D Advection averages 56–83, while Robertson ODE ranges from 16 to 67. The hard problems, Darcy Flow and 2D Navier–Stokes, defeat nearly all models, and only Claude passes gates on Darcy Flow with an average score of 49. Interestingly, all three models fail on the Navier–Stokes problem, but for distinct reasons. Gemini 2.5 Pro fails at compilation in all three runs, pass-ing incorrect argument types to DMStagCreate2d , for example supplying NULL 

where PETSc expects a PetscInt stencil width, indicating a fundamental mis-understanding of the DMStag API. Claude Opus 4.6 and GPT-5.2 both com-pile successfully but then crash at runtime. In five of the six runs, PETSc complains about “Matrix is missing diagonal entries,” indicating that the Ja-cobian assembly omits some grid points. One of GPT-5.2’s attempts fails with 

DIVERGED_NONLINEAR_SOLVE , implying a problematic nonlinear solver configu-ration. Overall, these behaviors indicate that this test case remains beyond the capabilities of current frontier models. 

6.4 Gate Failure Analysis 

Gate pass rates are 67% for Claude, 61% for Gemini, and 56% for GPT-5.2. Claude thus attains the highest pass rate. We observe that every program that compiles passes all the subsequent gates, indicating that the primary bottleneck is generating code that compiles against PETSc rather than subtle memory or API-compliance errors.

6.6 Qualitative Findings 

Our multi-dimensional evaluation highlights several informative cases. For the Vec/MPI Scatter problem, Claude triggers segmentation faults in three attempts despite otherwise high-quality code. On the stiff Robertson ODE problem, Gem-ini 2.5 Pro produces clean, well-structured code but poor numerical accuracy due to loose integrator tolerances it selects. Error handling scores are especially low (0.30 in 27 of 33 gate-passed attempts) since most models do not wrap PETSc calls in PetscCall() . This suggests that current LLMs have learned PETSc usage from older versions (pre-3.17) that used the legacy CHKERRQ() style. 

## 7 Discussion 

Why library-based evaluation matters. The dimensions that separate high from low scores (error-handling conventions, solver selection, and resource-management patterns) are inherently library-specific and do not arise in from-scratch implementations. By targeting PETSc, petscagent-bench evaluates the skill that matters for production HPC code: composing scalable building blocks rather than reimplementing known algorithms. Our results show that models struggle at two levels. The widespread use of the outdated CHKERRQ() 

macro points to training data staleness as a key bottleneck. In contrast, failures in assembling Jacobians for finite-element methods and in choosing tolerances appropriate to problem stiffness reflect deeper gaps in knowledge and reasoning about numerical methods and solvers. 

Implications for agent design. The fact that compilation errors dominate gate failures suggests multi-turn agents with compile–fix loops could substan-tially improve pass rates, as compiler diagnostics are often informative enough to guide corrections. The staleness we observe in error handling patterns indicates that enriching library-specific context for LLMs, e.g. with retrieval augmentation over up-to-date PETSc documentation, may be effective. Our framework sup-ports easy integration of new agents, providing a natural testbed for assessing whether these architectural improvements lead to better generated code. 

Limitations. The benchmark does not yet cover PETSc’s full breadth, such as GPU kernels and multiphysics, but can be easily extended. Also, LLM-based evaluators may vary across runs due to the inherent stochasticity of LLMs, though this is mitigated by temperature 0 and confidence weighting. 

## 8 Conclusion and Future Work 

We introduced petscagent-bench , an agents-evaluating-agents framework for AI-based scientific code generation that targets library-based HPC program-ming rather than from-scratch implementations. By making the evaluator a tool-augmented agent that compiles, executes, measures, and judges code while com-municating with the model under test via standardized protocols, the framework petscagent-bench 11 

enables multi-dimensional, black-box evaluation that goes beyond traditional pass/fail benchmarking. Our evaluation of Claude Opus 4.6, Gemini 2.5 Pro, and GPT-5.2 shows that frontier LLMs can sometimes generate readable, well-structured scientific code but fall short on correctness and library-specific metrics. Our findings highlight the value of multi-dimensional evaluation for scientific code generation. They could serve as baselines for future research on developing more effective coding agents, e.g., those with multi-turn interactions, retrieval augmentation, or tool use. For future work, we are expanding the benchmark to include GPU test problems.
