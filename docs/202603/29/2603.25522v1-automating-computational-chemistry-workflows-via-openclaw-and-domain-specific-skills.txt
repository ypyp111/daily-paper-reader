Abstract 

Automating multistep computational chemistry tasks remains challenging because reasoning, workflow specification, software execution, and high-performance computing (HPC) execution are often tightly coupled. We demonstrate a decoupled agent-skill design for computational chemistry automation leveraging OpenClaw. Specifically, OpenClaw provides centralized control and supervision; schema-defined planning skills translate scientific goals into executable task specifications;domain skills encapsulate specific computational chemistry procedures;and DPDispatcher manages job execution across heterogeneous HPC environments. In a molecular dynamics (MD) case study of methane oxidation, the system completed cross-tool execution, bounded recovery from runtime failures, and reaction network extraction, illustrating a scalable and maintainable approach to multistep computational chemistry automation. 

# Introduction 

Computational chemistry has become an indispensable tool for probing, predicting, and rationalizing chemical and materials systems. In practice, however, the major bottleneck is often not the lack of computational methods themselves, but the difficulty of reliably co-ordinating heterogeneous software tools, data representations, and execution environments for multi-step research tasks. Real applications routinely require molecular structure prepa-ration, input generation, parameter configuration, data-format conversion, remote job sub-mission, monitoring, error recovery, and downstream analysis across multiple codes and platforms. Because these operations are strongly interdependent, automation depends not only on executing individual tools, but also on maintaining consistency across task specifi-cation, software invocation, intermediate data handoff, and HPC execution. As such tasks become increasingly multi-step and data-intensive, the central challenge is no longer simply to automate isolated calculations, but to achieve robust, reusable, and auditable end-to-end automation. 1

2One major line of prior work approaches this problem through workflow-based automa-tion. Representative examples include general workflow infrastructures such as AiiDA, 2,3 

FireWorks, 4 and QMflows, 5 as well as domain-oriented workflow systems for specific scientific tasks. In artificial intelligence (AI)-driven atomistic simulation, for example, DP-GEN 6 and NepTrain 7 automate active-learning cycles for machine-learning potentials; CatFlow 8 and DP-TI 9 support free-energy and thermodynamic-integration workflows; ChecMatE, 10 Au-toSteper, 11 APEX, 12 and RiD-kit 13 provide workflow support for machine-learning-potential-based materials and molecular simulations; and mech2d 14 and maptool 15 illustrate high-throughput materials-screening workflows. These systems have proven highly effective for repetitive, well-defined, and high-throughput computational tasks by formalizing task de-pendencies, provenance tracking, data organization, and job execution across heterogeneous computing resources. However, workflow-based automation primarily relies on predefined task graphs and execution logic. Even when limited runtime dynamism is supported, these approaches become harder to extend when tasks require cross-software adaptation, context-dependent decision making, or recovery from unforeseen runtime failures. 4,16 

Alongside recent workflow centric automation frameworks for scientific model construc-tion and training, which demonstrate the value of modular tool integration and iterative execution in domain specific settings, 17 large language model based agent systems have opened a new direction for scientific automation. By interleaving natural language reason-ing with external tool use, intermediate state inspection, and task level replanning, these systems can adapt to evolving execution contexts in ways that are difficult to hard code in predefined workflows. 18–20 Recent agent studies related to computational chemistry au-tomation, including ChemCrow 21 and CACTUS, 22 suggest that large language model based systems can support computational chemistry workflows by coordinating domain tools and scientific constraints during execution. 23 This emerging direction is important because it moves automation beyond fixed procedural pipelines and toward execution strategies that can respond to intermediate results, tool outputs, and changing task context. 21–23 

3Current approaches to computational chemistry automation occupy two largely sepa-rate design regimes. Workflow systems provide explicit execution structures and, in some cases, robust scheduler integration, but they typically rely on predefined control flow and limited recovery semantics. 2,4,5,16 Computational chemistry focused agentic systems provide greater runtime flexibility through dynamic reasoning and tool selection, yet workflow struc-ture, recovery logic, tool-routing policies, and execution-environment assumptions are often embedded directly within specialized agent stacks. 21,22,24 As a result, extending capability may require redesigning agent-side orchestration rather than simply replacing an executable module. These contrasts can be examined in terms of workflow explicitness, recovery model, HPC grounding, module-replacement granularity, and dependence on computational chem-istry specific design. In this view, the central limitation is not insufficient reasoning capa-bility, but engineering entanglement: the tight coupling of reasoning, workflow specification, domain execution, and infrastructure adaptation, which prevents them from being indepen-dently replaced, extended, and maintained. More broadly, current automation systems can be viewed along a spectrum from scripted workflows, to workflows with LLM-enabled nodes, to specialized agents whose task logic is largely embedded in prompts and tool-routing policies. Across this spectrum, reusable tool access does not by itself provide task-specific execution competence: a general agent may have access to broad capabilities through runtime-loaded tools or skills, yet still require an explicit task specification for how those capabilities should be combined to achieve a concrete computational task objective. At the same time, computational chemistry places unusual pressure on software-facing execution layers, because executable substrates change frequently across software packages, file formats, and HPC environments. These considerations moti-vate a decoupled design in which general-purpose control is separated from computational chemistry execution. Rather than embedding domain procedures directly into a specialized agent stack, we organize automation around a general-purpose agent that handles coordi-nation, state tracking, and decision making, while delegating concrete scientific operations 4to reusable skills. In such a formulation, new domain capabilities are introduced by adding or replacing executable skills rather than redesigning the agent itself, which is especially attractive for computational chemistry. Here, we adopt this idea in the form of an OpenClaw-grounded decoupled agent-skill design for computational chemistry automation. OpenClaw 25 provides the general-purpose control and coordination mechanism; schema-defined planning skills convert scientific goals into executable task specifications; domain skills encapsulate computational chemistry pro-cedures and software-facing operations; and DPDispatcher 26 supplies a scheduler-aware ex-ecution interface for heterogeneous HPC environments. In this way, new computational chemistry capabilities can be introduced, replaced, and reused at the skill level without redesigning the agent itself. In this work, we demonstrate this design through a case study on methane oxidation re-action exploration, adapted from a prior neural network-based molecular dynamics study of combustion chemistry. 27 Starting from a natural language instruction, the system constructs an explicit task specification and carries out molecular optimization, file format conversion, initial reactive system construction, molecular dynamics simulation, HPC dispatch and mon-itoring, failure recovery, and trajectory analysis. This case illustrates that computational chemistry procedures can be externalized into reusable skills, providing a practical basis for computational chemistry automation. To support reuse and reproducibility, we have also released an open-source library for computational chemistry skills, covering quantum chemistry, molecular dynamics, machine-learning potentials, and molecular representation. 

# Methodology and Architecture 

# Design Principles and System Organization 

Our design is motivated by a simple principle: computational chemistry automation should externalize not only reusable computational procedures, but also the task-specific 5logic that determines how those procedures are organized and executed. In our formula-tion, reusable skills provide domain operations together with usage descriptions available at runtime, while a task manifest converts a computational chemistry task description into an explicit specification of which skills to combine, in what order, and under what conditions. Under this design, the agent does not require computational chemistry specific fine tuning or a deeply specialized internal policy. Instead, domain appropriate behavior emerges from the runtime combination of a general agent, reusable skills, and task specific guidance. We instantiate this design using OpenClaw as the general agent framework, augmented with reusable skills for task coordination and computational chemistry execution, as shown in Figure 1. Rather than constructing a specialized computational chemistry agent from scratch, we build on a general agent framework maintained by a broader software community, thereby improving software sustainability while allowing domain experts to focus on task specific capabilities. At the same time, a general agent alone is insufficient for computational chemistry automation, because it does not natively encode scheduler semantics, scientific data formats, or software-specific execution procedures. We therefore place these scientific and execution related capabilities in skills that are loaded at runtime and can be revised independently of the core agent. To make the decoupled skill architecture practical for computational chemistry automa-tion, the execution layer must support isolated deployment and reproducible tool invocation. Most skills in our ecosystem are therefore built around the uv toolchain. In particular, uvx invokes tools in temporary isolated environments without requiring global installation, which reduces dependency conflicts and keeps the host system clean. This allows each skill to be represented as an executable command together with its required packages, so that depen-dencies can be resolved at runtime while remaining modular at the skill level. Because uv also caches previously built dependencies, subsequent executions can reuse local packages, improving deployment efficiency and supporting reproducible workflow execution. Skills can further constrain tool resolution to specific packages or explicit versions, thereby improving 6consistency across runs. Taken together, these features provide isolated and reproducible execution environments for skills without entangling dependency management with the core agent. The role of skills is not to make decisions or act autonomously, but to serve as structured capability packages containing instructions, templates, and validation logic. When a request matches a skill, OpenClaw loads its content into the model context. The language model then reasons over the user goal, the current state, the loaded skills, and the execution feedback to determine the next action, which OpenClaw then executes. Tool outputs, generated files, and external program feedback are returned to this central control loop to inform the next decision step. Consequently, high level planning, HPC submission, and computational chemistry operations are all orchestrated by OpenClaw rather than performed independently by isolated modules. The skills, in turn, contribute capability descriptions and procedures that the central loop can invoke, inspect, and revise. From this perspective, our architecture is best described as one general agent framework with an extensible skill ecosystem. Some skills focus on workflow coordination, some on execution grounding, and some on domain operations, but they all enter the system through the same mechanism of context loading, model reasoning, tool execution, and feedback. This design combines the maintainability of a general agent framework with the task specificity required for computational chemistry. 7Figure 1: Architecture of the OpenClaw-based decoupled agent-skill framework for compu-tational chemistry automation. 

# OpenClaw Control Loop and Skill Invocation 

At runtime, OpenClaw maintains session context that includes the user request, tool ac-cess, and state tracking across long tasks. For each step, it first reads the current conversation state together with the outputs of previous actions. The language model then produces the next action, which may involve loading a skill that matches the current need, asking the user a question, calling a tool, generating a workflow artifact, or executing a domain-specific script. OpenClaw validates and executes that action, captures standard output, error out-put, generated files, and tool results, and appends them to the active context for the next turn. This loop continues until the task is completed or human intervention is required. This process is illustrated schematically in Figure 2a. The detailed logical flow of the automated execution engine is shown in Figure 2b. Upon task initiation, the system identifies and retrieves the necessary skill sets. This is followed by a task-planning phase in which the LLM decomposes the high-level request into a series of executable jobs. During the execution stage, the system handles job submission to the remote environment and monitors its status. A key feature of this workflow is the autonomous error-recovery loop: when execution does not reach a successful state, the agent retries by 8Figure 2: (a) Schematic of the LLM-based decision-making process; (b) Workflow of the automated computational task execution process. analyzing execution logs and, if necessary, requesting user guidance to resolve the bottleneck. This cycle repeats for each sequential task within the workflow manifest until all dependencies are satisfied, ultimately generating a final summary of the research outputs. Progressive skill loading is particularly advantageous in computational chemistry; it keeps the context window focused on the immediate task, thereby avoiding the overhead of process-ing documentation for every supported software package simultaneously. This modularity also facilitates system expansion, as integrating new software typically requires only adding or updating specific skills rather than modifying the core agent logic. Such decoupling motivates the adoption of a general-purpose agent framework over an architecture tailored exclusively to computational chemistry. Under this design, the core infrastructure lever-ages broader software maintenance and community development, while scientific expertise remains concentrated within reusable, domain-specific skills. 9The Agent Taskboard Manifest and the DPDispatcher Skill 

Two skills are especially important in our implementation. The first converts an open-ended computational task description into an explicit, executable workflow representation, thereby bridging the gap between broad capability access and task-specific execution com-petence, whereas the second grounds long-running tasks in real execution environments. The Agent Taskboard Manifest skill addresses a key limitation of purely tool-augmented general agents. While dynamic skill loading endows the model with a wide range of ex-ecutable capabilities, tool access alone does not specify how those capabilities should be ordered and combined to fulfill a concrete computational task description. In traditional dedicated agents, such task-specific execution competence is typically embedded directly in the agent through carefully engineered prompt constraints that channel the model along a fixed procedural path. To retain generality without compromising execution rigor, we instead implement workflow coordination as a specialized skill. This skill serves as a meta-prompting layer that generates task-guidance prompts and execution constraints dynamically at run-time. Specifically, it translates the user-provided computational task description into a struc-tured task specification that explicitly defines stages, dependencies, required inputs, ex-pected outputs, and validation conditions. To control reasoning complexity and manage the language model context window, the resulting workflow description adopts a lazy-loading strategy in which only the currently active subtask is introduced into the agent context. This explicit and modular representation improves inspectability, facilitates state-dependent recovery, and provides a general agent with the targeted procedural constraints that are otherwise hardcoded in dedicated expert systems. Because the coordination mechanism is dynamically loaded at runtime in the same manner as other skills, it remains replaceable and can be refined without modifying the underlying OpenClaw runtime. The DPDispatcher skill implements execution grounding. DPDispatcher, which we de-veloped with collaborators as an open-source package, generates job scripts, submits jobs, 10 monitors their status, and retrieves result files across local and remote environments. 26 In our framework, the corresponding skill translates agent intent into validated DPDispatcher descriptors, including machine settings, resource requests, file-transfer specifications, and task commands. This design shields the core agent from scheduler-specific syntax while still exposing the operational choices that matter for scientific execution. Through DPDis-patcher, the same agent can target local shell execution, Slurm, 28 the Portable Batch System (PBS), 29,30 and Load Sharing Facility (LSF) 31 through a unified interface. Queueing, wait-ing, monitoring, and result collection are therefore treated as normal workflow states rather than as ad hoc shell-side bookkeeping. This point is important in computational science, where the practical bottleneck is often not a single command, but the full life cycle of remote execution.

# Case Study: Methane Oxidation Reactive MD Automa-tion 

To evaluate the practical feasibility of the proposed architecture for computational chem-istry workflows, we apply it to a case study involving reactive molecular dynamics simulation of methane oxidation and subsequent extraction of reaction pathways from the resulting tra-jectories, as shown in Figure 3. This task is adapted from a prior molecular dynamics study of combustion chemistry based on a neural network potential, 27 and is used here to examine whether a general agent equipped with reusable planning and domain skills can orchestrate the corresponding workflow in an explicit, traceable, and recoverable manner. The case is suitable for this purpose because it requires coordination across multiple interdependent stages, including molecular structure preparation, quantum chemical optimization, mixed-system construction, reactive molecular dynamics simulation, and trajectory analysis, while also depending on interoperability across heterogeneous software packages and execution environments. In this case, we submit a natural-language request to OpenClaw, with GPT-5.2 (xhigh) or GPT-5.4 (xhigh) serving as the reasoning engine. The request defines a multistep methane 12 oxidation workflow that includes generating isolated-molecule structures with Open Babel, 35 

optimizing CH 4 and O 2 at the B3LYP 55 /6–31G(d,p) 56,57 level of theory, converting the op-timized structures into standardized molecular files with dpdata, 36 packing a bulk system containing 50 CH 4 and 100 O 2 molecules at a target density of 0.25 g/cm 3 using Pack-mol, 37 and generating a LAMMPS-readable data file. For the reactive molecular dynamics stage, the agent receives as external input a pretrained Deep Potential 58 model for the methane oxidation system, here taken from Ref. 59, and incorporates it into LAMMPS 43 

through DeePMD-kit. 39,40 It then performs short structural relaxation or low-temperature pre-equilibration to remove unfavorable local contacts, heats the system to 3000 K under the NVT ensemble with the Nose–Hoover thermostat, and runs a 1 ns reactive molecular dynamics simulation with a time step of 0.1 fs while recording key simulation parameters, thermodynamic outputs, and atomic trajectories. Finally, ReacNetGenerator 33 analyzes the trajectory to track CH 4, O 2, radical intermediates, and oxidation products, and to extract the dominant methane oxidation pathways and key intermediates. Rather than mapping this computational task description directly to a fixed procedure, the framework invokes a schema-defined task-planning skill to externalize the objective into an explicit workflow manifest. Rather than serving as a simple step list, this manifest encodes stage dependencies, completion criteria, validation conditions, and execution constraints in a form that can be interpreted at runtime. For the present case, the resulting computational workflow comprises six stages: molecular preparation, geometry optimization, file format conversion, reactive system construction, molecular dynamics simulations, and trajectory analysis. Pre-flight validation, scheduler submission, and runtime monitoring. Throughout this process, OpenClaw maintains the global task state and orchestrates the execution flow, while reusable domain skills provide the computational chemistry procedures required at each stage. As a result, execution is governed by validated intermediate outputs rather than by a fixed linear script. 13 Figure 3: Workflow of the methane oxidation MD simulation. Before intensive execution, the framework performs pre-flight checks on executables, de-pendencies, scheduler access, and required file paths, thereby reducing failures arising from environmental mismatches. Reactive MD execution is then grounded in the target HPC envi-ronment through the DPDispatcher 26 skill, which translates the planned task into scheduler-compatible job descriptors. In this way, the orchestration environment and the scientific execution environment remain explicitly connected while operationally separated. During execution, task advancement is state-driven rather than purely sequential. Stage transitions occur only after the required intermediate outputs have been produced and val-idated. If an individual stage fails, the framework inspects the current execution state and applies stage-appropriate bounded recovery actions—such as parameter repair, retry, or rollback—before proceeding. When recovery conditions are not satisfied, execution halts, thereby preventing unbounded deviation from the intended task procedure. We successfully executed the entire workflow in repeated end-to-end demonstrations. Owing to the inherent stochasticity of LLMs, individual runs exhibited variations in the dialogue trajectories, the generated action plans, and the specific errors encountered. In these demonstrations, the system leveraged error feedback to resolve such issues and complete the assigned tasks. 

# Conclusions 

This work demonstrates an OpenClaw-grounded decoupled agent–skill design for mul-tistep computational chemistry automation. OpenClaw provides general-purpose task co-14 ordination and supervision, planning skills externalize scientific goals into executable task specifications, the DPDispatcher-based submission skill grounds computation in real HPC environments, and domain skills provide software-specific procedures and result-handling logic within the agent control loop. Through the methane-oxidation reactive MD case study, we show that this framework can organize multistage computational chemistry tasks, co-ordinate cross-tool execution, and support bounded recovery from runtime failures without relying on a fully pre-encoded workflow. These results suggest that general-purpose agents, coupled with modular skill ecosystems, offer a scalable and maintainable path toward prac-tical automation across computational chemistry and materials science. Future work will focus on expanding the reusable skill ecosystem to support a broader range of computational tasks—such as high-throughput materials screening, periodic solid-state computations, and active-learning potential generation—while fostering community contributions to maintain the library. Furthermore, integrating the agent with rigorous provenance-tracking databases and enabling closed-loop autonomous workflows remain important directions for scientific discovery.
